# LOL 战队论据生成器 —— 框架与需求文档 v0.1

名称：Rift pettifogger/ 赢学联盟

## 一、项目定位

一句话:输入战队 A 和 B,工具在真实比赛数据中搜索"A 比 B 强"的论据链,并生成带出处、可直接粘贴到贴吧/NGA 的论证文本。

明确的非目标:不做客观实力排名,不做 Elo 或任何评分系统。允许 A>B 与 B>A 在同一份数据下同时成立——本工具是辩护律师,不是法官。

产品形态:纯静态网站,托管于 GitHub Pages;数据由 GitHub Actions 定时抓取并提交为 JSON;无后端、无数据库、零运维成本。

## 二、核心术语

- **战队实体**:以品牌为单位,换血不换队。改名与收购通过别名表归并到同一实体(如 DWG → DAMWON Gaming → DWG KIA → Dplus KIA)。
- **Series(大局)**:一次 BoX 对阵整体。**Game(小局)**:其中的单场游戏。
- **赛事级别**:普通赛事 / 国际赛(First Stand + MSI + Worlds)/ 仅 Worlds。
- **证据(边)**:某条规则在当前过滤器下推出的有向关系 A→B,附带完整出处。
- **论证链(路径)**:证据图中从 A 到 B 的一条路径。

## 三、全局过滤器(用户可调,影响所有规则)

| 过滤器 | 默认值 | 用途 |
|---|---|---|
| 起始时间 | 不限 | "只看换血后的新阵容" |
| 截止时间 | 今天 | "冻结在 S13 结束时判案" |
| 赛事范围 | 全部 | 全部 / 国际赛 / 仅 Worlds |
| 大局 / 小局 | 大局 | 规则 3 的统计口径开关 |
| 邻近窗口 | 90 天 | 规则 2 中两场对阵允许的最大时间差 |
| 跨赛制严格模式 | 关 | 见规则 2 |

全部过滤器状态编码进 URL 参数——判案结果一条链接即可复现,直接甩到帖子里。

## 四、规则引擎

### 规则 1:直接交手

时间窗内双方**最近一次** series,胜者获得一条边。Bo1 也计入(直接击败不需要比小分)。更早的交手记录不走这条规则,交给规则 3 消化。

证据展示:日期、赛事、赛制、比分、原始链接。

### 规则 2:共同对手

前提:存在 C,使 A、B 都与 C 打过 series,且两场 series 时间差 ≤ 邻近窗口(同一赛事内自动视为满足)。Bo1 一律排除。只处理"同胜"或"同负"——一胜一负的情况本质是 A>C>B,由规则 1 + 规则 4 自动覆盖,不在此重复。

**同赛制**:直接比比分。同胜者丢局少的强,同负者取局多的强,比分完全相同不产边。

**跨赛制(Bo3 vs Bo5)**:引入表现档位:

| 结果 | 档 0 | 档 1 | 档 2 |
|---|---|---|---|
| 负 | 被横扫(0-2 / 0-3) | 拿过局但未进决胜局(1-3) | 打满决胜局(1-2 / 2-3) |
| 胜 | 决胜局险胜(2-1 / 3-2) | 有丢局未打满(3-1) | 横扫(2-0 / 3-0) |

跨赛制仅当档位**严格不同**时产边,高档位者强。开启严格模式后要求档位差 = 2,即只承认"横扫 vs 决胜局"这种一目了然的对比(精确对应"0-2 负 vs 2-3 负"的原始例子)。

同一对 (A, C) 或 (B, C) 在窗口内有多次交手时,每个满足条件的配对都可产生候选证据,展示时优先取两场时间最邻近的一对。

### 规则 3:历史战绩

时间窗 + 赛事范围内 A vs B 的全部交手,A 胜率**严格大于 2/3** 则产边。N ≥ 1 即可成立(1 战 1 胜 = 100%)。

统计口径:默认按大局(series 胜负),用户可切换为小局(game 胜负)。小局数直接由比分推得,无需逐局数据。Bo1 在大局口径下计为 1 个 series。

**范围自动降级**:若当前范围下不产边,依次收窄尝试 国际赛 → 仅 Worlds,第一个通过的范围产边,并在证据卡上强制标注统计口径(如"仅 Worlds 交手:2-0")。标明口径的选择性统计是贴吧判案的正当手段,但必须标明。

### 规则 4:传递链

在规则 1–3 生成的有向图上做 BFS,路径不允许重复经过节点,**不限长度**,按(长度升序,链强度降序)排序,默认展示前 5 条,可展开全部。

链强度 = 最弱一环的强度。单边强度排序:规则 1 > 规则 2 > 规则 3;同规则内时间越近越强。强度只用于排序展示,不做淘汰——只要存在路径,哪怕八连传导也给结果,嘴硬到底是用户的权利。

找不到任何路径时:明确告知,并提示放宽哪个过滤器可能有戏。

### 反方视角

每次判案同时反向搜索 B>A 的论据,折叠展示为"对方可能这样反驳你"。预判反驳是判案工具的核心竞争力。

## 五、数据层

**数据源**:Leaguepedia(lol.fandom.com)Cargo API 为主。所需表:
- `MatchSchedule`:series 级数据(队伍、比分、赛制、日期、所属赛事)——核心表,v1 只需要它。
- `Tournaments`:赛事元数据(级别、赛区),用于打国际赛 / Worlds 标签。
- `TeamRedirects`:官方别名归并数据,作为别名表基础。

备选与校验:Oracle's Elixir 的公开 CSV。

**采集范围(v1)**:四大赛区联赛(LPL、LCK、LEC、LCS/LTA)+ 三大国际赛。国际赛场次天然包含小赛区队伍,因此入围赛传递链可用;小赛区联赛内部数据留待扩展。

**数据模型**(JSON,按年份分片):

```
series:  { id, date, tournament, tier, best_of, t1, t2, s1, s2, flags }
teams:   { canonical_id, display_name, aliases[], region }
index:   { years[], counts, generated_at }
```

flags 用于标注弃权、重赛等特殊情况。v1 不需要逐局(game 级)数据——小局口径由 s1/s2 即可推算;逐局数据留给未来的"对位/BP 论据"扩展。

**更新机制**:GitHub Actions 每日 cron 跑 Python 抓取脚本,增量更新当年 JSON 并 commit;提供 workflow_dispatch 手动全量重建。注意 Cargo API 的分页与限速,脚本需带指数退避。

**体量预估**:四大赛区 + 国际赛自 2013 年至今约 1.5–2.5 万条 series,分年 JSON 合计 3–5 MB,gzip 后不足 1 MB,前端全量加载后在内存建图,毫无压力。

## 六、前端

技术选型:纯静态,vanilla JS/TS + Vite 即可,不强制框架。所有规则计算与图搜索在浏览器端执行(边依赖过滤器状态,必须运行时生成;可按"对手 → 交手记录"预建索引加速规则 2)。

页面流程:选 A → 选 B(带别名模糊搜索的选择器)→ 调过滤器 → 判案 → 论据列表。

**论据卡片**:链路示意(A —规则1→ C —规则3→ B)+ 每一环的日期、赛事、赛制、比分、口径标注、Leaguepedia 原始链接。

**一键复制**:生成论坛可直接粘贴的纯文本,含全部出处。论据的杀伤力在于可验证。

页脚免责声明:娱乐工具,论据 ≠ 真理,输出结果与对立结论可能同时存在。

## 七、边界情况清单

- **Bo2 平局(1-1)**:大局口径下剔除出规则 3 样本,小局口径正常计入;规则 1、2 中平局不产边。
- **弃权(FF)、重赛、Chronobreak**:以 Leaguepedia 官方结果为准计入,证据卡标注"含弃权"。
- **表演赛 / 全明星**:不入库。
- **别名遗漏 / 马甲队**:以 TeamRedirects 为基础,另设 overrides.json 手工覆盖文件兜底。
- **A = B**:拒绝输入。
- **A>B 与 B>A 同时成立**:正常现象,正反双方都展示。
- **时间窗内某队无任何比赛**:提示无数据并建议放宽时间窗。

## 八、开发里程碑

1. **M1 数据管道**:抓取脚本 + Actions 工作流 + JSON 产出,重点验证别名归并正确性(抽查 DK、Gen.G、T1 等改名大户)。
2. **M2 规则引擎**:规则 1–3 实现为纯函数,手工构造用例做单元测试,重点覆盖跨赛制档位、Bo2、范围降级。
3. **M3 图搜索**:边生成、BFS、排序、反方搜索。
4. **M4 UI**:选择器、过滤器、论据卡、复制导出、URL 分享。
5. **M5 扩展**:"嘴硬模式"(自动搜索能让 A>B 成立的过滤器组合并直接推荐)、小赛区联赛数据、逐局数据。

## 九、已替你拍板、随时可推翻的决策

1. 跨赛制档位表是对"0-2 负 vs 2-3 负"例子的推广(例如 0-2 负 vs 1-3 负也产边);嫌宽就开严格模式,只认档位差 = 2。
2. Bo2 平局:大局剔除、小局计入。
3. 规则 1 定义为"最近一次交手的胜者",而非"曾经胜过";旧胜场由规则 3 体现。
4. 规则 3 范围降级取"从宽到窄第一个通过的范围"。
5. v1 数据范围为四大赛区 + 三大国际赛,小赛区队伍仅通过国际赛场次入图。
